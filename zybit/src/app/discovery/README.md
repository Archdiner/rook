# /discovery — Phase 0 Customer Discovery Survey

This is the customer discovery survey that powers our Phase 0 outreach to the Cornell entrepreneurship mailing list. It lives at the route `/discovery` and is intentionally **not** linked from the landing page, the navigation, the footer, or any other public surface — the URL is meant to be sent directly in outbound emails so we control distribution and can attribute responses to specific outreach batches. The page asks eight questions about how the recipient currently approaches conversion, activation, and retention work, and gates the bulk of the survey behind a Yes/No screener so people who haven't done relevant work in the last 90 days can opt out gracefully without filling out anything else. Voice and aesthetic match the landing (cream `#FAFAF8`, ink `#111`, Inter for display, Newsreader for prose) but stripped of motion and 3D — it should read as a calm, well-typeset document, not a marketing page.

The data model for each submission is a single JSON object with `id` (UUIDv4), `submitted_at` (ISO 8601), `q1`–`q8`, and the conditional fields `q3_other`, `q4_note`, `q7_email`, `q8_url`. On every successful POST to `/api/discovery`, we send a structured email to `sar367@cornell.edu` via Resend (the source of truth) and append the same record as a single line to a Vercel Blob at `discovery-responses/${YYYY-MM}.jsonl` using read-modify-write with `allowOverwrite: true` and `addRandomSuffix: false`. To download and analyze a month of responses, run `vercel blob download discovery-responses/2026-04.jsonl | jq` from the linked Vercel project (or fetch the public blob URL directly). The blob write requires `BLOB_READ_WRITE_TOKEN` — if that env var is missing, the API logs a warning and skips the blob silently so we never lose a response. **TODO (Phase 1):** migrate this storage to Neon Postgres when the database lands so we can run analytical queries directly, retain referential integrity with users created in the app, and stop relying on JSONL append semantics.
